# Kiro 前端设计文档

> 版本：A14 前端体验改造草案
> 日期：2026-07-05
> 范围：先改 Web 主聊天区，再逐步扩展到记忆、画像、设置页面。

---

## 1. 设计目标

Kiro 的前端不是普通聊天壳，也不是调试 Dashboard。它需要同时承担三件事：

1. **让聊天保持轻松自然**：主界面首先是对话，不应被大量记忆字段、工具名和调试信息压住。
2. **让记忆系统可理解**：用户能看见“记忆有没有工作”，但不需要理解每个底层实现细节。
3. **为多端使用做准备**：Web、PWA、后续 App 需要共享一套信息架构，尤其是 session、记忆状态、模型切换和语音交互。

本轮改造先从主聊天区开始，因为它是最高频入口，也是语音、文件、表情、记忆状态和多端同步最容易互相打架的地方。

---

## 2. 设计原则

### 2.1 主聊天区优先

主聊天区只展示当前对话需要的信息：

- 当前 AI 名称与模型状态。
- 当前会话内容。
- 输入区。
- 可折叠的记忆状态入口。

不要在主聊天区常驻展示大量技术状态，例如完整 Gateway 注入、全部工具 JSON、完整候选队列。这些内容应进入右侧抽屉或二级页面。

### 2.2 记忆可见，但不打扰

记忆状态应该像“状态灯”，而不是“报告书”。

主聊天区只需要看到：

- 记忆状态是否正常。
- 当前 session 是否同步。
- 是否有候选需要处理。
- 是否有维护异常。

详细内容进入右侧记忆抽屉或“记忆”页面。

### 2.3 语音入口回到输入区

此前语音按钮独立放在输入区外，容易和发送按钮、工具按钮重复。新方案把语音放回 composer 内：

- 输入框左侧：表情、附件。
- 输入框右侧：语音模式、发送。
- 语音按钮不再作为一个大圆形悬浮入口。
- 语音和发送是两个不同动作：语音用于说话，发送用于提交当前文本。

### 2.4 调试能力分层

普通用户界面与高级调试界面要分层：

- 主聊天区：只展示可读状态。
- 右侧记忆抽屉：展示运行状态和轻量控制。
- 记忆页：查看记忆、候选审批、维护入口。
- 设置页：模型、同步、语音、记忆策略、安全。
- Dashboard：保留高级调试和底层细节。

## 3. 视觉系统

### 3.1 色彩

| 名称 | 色值 | 用途 |
| --- | --- | --- |
| Warm ivory | `#fff8f4` | 页面背景，保持温暖但不厚重。 |
| Kiro coral | `#ff6b4a` | 主操作、选中态、重要提示。 |
| Soft peach | `#ffe1d5` | 用户气泡、轻提示背景。 |
| Memory teal | `#5fb99a` | 在线、稳定、同步正常。 |
| Dream lavender | `#a78bfa` | Dream、内在状态、辅助状态。 |
| Ink brown | `#2a211d` | 主文字。 |
| Muted brown | `#8b6f63` | 次级文字、说明。 |
| Line peach | `#f3cdbf` | 边框、分隔线。 |

避免大面积渐变、装饰圆球和纯视觉噪音。Kiro 的温暖应来自细节、留白和文本节奏，而不是大面积粉色堆叠。

### 3.2 字体与字号

当前可先使用系统字体，后续再决定是否引入字体文件。

| 层级 | 建议 |
| --- | --- |
| 页面标题 | 22-26px / 700 |
| 模块标题 | 15-17px / 700 |
| 正文 | 14-15px / 400-500 |
| 辅助说明 | 12-13px / 400 |
| 标签与状态 | 11-12px / 600 |

原则：主聊天气泡里的文字不应过小；右侧状态抽屉可以更密，但必须可扫读。

### 3.3 圆角与阴影

- 常规卡片圆角：8px。
- 输入框圆角：10-12px。
- 按钮圆角：8-999px 视形态而定。
- 不使用层层套卡。页面区块应是平整布局，卡片只用于具体信息项。

## 5. 第一阶段：主聊天区改造

### 5.1 目标

第一阶段只改对话页，不重做所有页面。

需要完成：

1. 固定左侧导航。
2. 中间聊天区保持干净。
3. 右侧记忆状态抽屉可打开/关闭。
4. 输入区重构，加入表情、附件、语音、发送。
5. 接入现有 `window.KiroMemoryRuntime` 和 `window.KiroMemoryAPI`，但只展示必要状态。

暂不做：

- 完整记忆库列表。
- 完整画像编辑。

## 6. 主聊天区模块说明

### 6.1 左侧导航

保留四个入口：

| 入口 | 用途 |
| --- | --- |
| 对话 | 默认首页，聊天与轻量记忆状态。 |
| 记忆 | 查看长期记忆、候选、词图、维护。 |
| 画像 | 查看和审批用户画像、Kiro 状态、关系天气。 |
| 设置 | 模型、语音、多端同步、记忆策略、安全。 |

底部保留 Kiro 状态小卡，但内容压缩：

```text
Kiro 陪伴中
关系天气：稳定
Dream：已更新
维护：正常
```

### 6.2 顶部栏

顶部栏展示：

- Kiro 头像与名称。
- 当前模型：Gemini / GPT / Claude / GLM。
- 在线状态。
- 「记忆状态」按钮。
- 主题/更多菜单。

模型切换只在顶部展示当前状态，详细配置放到设置页。

### 6.3 聊天气泡

聊天气泡要减少“舞台感”，更接近工作中的长期伙伴对话。

AI 气泡：

- 白底。
- 左侧小头像点。
- 支持建议动作 chip。
- 长回复要有清晰段落，不要挤在大块文本里。

用户气泡：

- 浅珊瑚背景。
- 右侧对齐。
- 时间和发送状态放在气泡右下。

### 6.4 快捷动作

快捷动作只出现在 AI 回复后，数量控制在 2-3 个。

示例：

- `整理 MVP 清单`
- `看时间线`
- `打开记忆状态`

不要把所有系统工具都放成快捷按钮。

## 7. 输入 Composer 改造

### 7.1 新结构

```text
┌──────────────────────────────────────────────────────┐
│ 🙂  📎  和 Kiro 聊点什么...        🎙  语音   ➤ │
└──────────────────────────────────────────────────────┘
```

左侧：

- 表情按钮。
- 图片 / 文件按钮。

中间：

- 多行文本输入。
- 最大高度限制，超过后内部滚动。

右侧：

- 语音按钮。
- 发送按钮。

### 7.2 语音按钮交互

语音按钮有两种状态：

| 状态 | 行为 |
| --- | --- |
| 默认 | 点击开始录音或按住说话，显示录音状态。 |
| 录音中 | 按钮变为停止/发送语音态，输入区显示录音时长。 |

建议第一版先做点击开始/再次点击结束，后续再加按住说话。

### 7.3 语音按钮位置调整理由

语音属于输入方式，不属于全局动作。它应该和文本输入、文件、表情放在同一组里，而不是悬浮在输入区外。

这样可以避免：

- 页面底部出现两个主要动作按钮。
- 用户分不清“语音”和“发送”的优先级。
- 移动端底部空间被大按钮占满。

## 8. 右侧记忆状态抽屉

### 8.1 展示原则

右侧抽屉不是 Dashboard。它只回答：

1. Kiro 现在有没有被正确唤醒？
2. 当前 session 是否同步？
3. 主动工具是否可用？
4. Dream / 维护是否正常？
5. 短期上下文配置是否合理？

### 8.2 模块

| 模块 | 内容 |
| --- | --- |
| 醒醒锚点 | 新窗口用于接续的 1-2 条轻量锚点。 |
| 当前会话 | session_id、同步状态、最近历史数量。 |
| 主动工具 | MCP-compatible 工具状态，不展示底层 JSON。 |
| Dream | Digest / Crystallization 是否更新。 |
| 维护 | 向量、词图、归档预览是否正常。 |
| 短期上下文设置 | 轮数、字符上限，可重置。 |

### 8.3 不展示内容

右侧抽屉不展示：

- Darkroom note 正文。
- 完整 system prompt。
- 完整 Gateway JSON。
- 未经审批的完整画像推断。
- 大段长期记忆正文。

## 9. 后续页面方向

### 9.1 记忆页

记忆页负责管理长期记忆。

优先内容：

- 记忆库列表。
- 类型筛选：全部 / 长期 / 动态 / 计划 / 归档。
- 搜索。
- 候选审批。
- 词图概览。
- 维护入口：重建词图、重建向量、归档预览。

### 9.2 画像页

画像页负责区分事实、状态和候选。

优先内容：

- 用户画像：只放已确认事实。
- Kiro 状态：I State、响应风格、当前 mood。
- 关系天气：作为 post-reply state，不作为用户事实。
- 最近连续性：上一个 session / 当前项目。
- Profile Fact 候选审批。

### 9.3 设置页

设置页负责配置。

优先内容：

- 模型与路由。
- 语音与输入。
- 多端同步。
- 记忆策略。
- 安全与维护。

## 10. 实施顺序

### P1：主聊天区骨架

1. 重构页面三栏布局。
2. 保留现有聊天发送、语音发送、历史加载能力。
3. 移除独立大语音按钮。
4. 重建 composer。
5. 加入右侧记忆状态抽屉静态结构。

### P2：接入运行状态

1. 右侧抽屉读取 `window.KiroMemoryRuntime`。
2. 展示 session、Pulse、MCP manifest、Dream、Maintenance。
3. 支持刷新状态。
4. 支持短期上下文轮数 / 字符配置。

### P3：主聊天细节

1. 优化消息气泡宽度、间距、时间位置。
2. 优化 loading 状态。
3. 优化错误提示。
4. 支持文件/图片按钮的空状态提示。
5. 移动端适配。

### P4：记忆页

1. 先做只读记忆列表。
2. 再做候选审批。
3. 最后做维护操作入口。

### P5：画像页与设置页

1. 画像页先只读，再审批。
2. 设置页先做模型、同步、语音，再做高级记忆策略。

## 11. 第一阶段验收标准

主聊天区第一阶段完成后，需要满足：

- 主聊天区视觉接近设计稿方向。
- 语音按钮在 composer 内，不再重复出现。
- 表情、文件、语音、发送四个入口位置明确。
- 右侧记忆状态可打开/关闭。
- 手机端不常驻右侧栏，改为抽屉或底部面板。
- 文本聊天、语音聊天、session 历史加载不回退。
- 不暴露 Darkroom 正文。
- 不把调试 JSON 直接展示给普通用户。

---

## 12. 需要避免的问题

1. 不要把 Dashboard 搬进主聊天页。
2. 不要让记忆状态比聊天更抢眼。
3. 不要再做多个语音入口。
4. 不要让右侧抽屉默认占用移动端空间。
5. 不要把 Relationship Weather 写得像用户事实。
6. 不要让设置项过早堆满主界面。
